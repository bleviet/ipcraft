# Custom Templates

IPCraft's `generate` command uses Jinja2 templates to produce VHDL, vendor
files, and testbenches. The built-in template set covers the standard output
layout, but you can override individual templates — or supply a complete
alternative methodology — without modifying the package.

---

## Built-in Templates

All built-in templates live in `ipcraft/generator/hdl/templates/`:

| Template file | Generated output |
|---------------|-----------------|
| `package.vhdl.j2` | `rtl/{name}_pkg.vhd` — types, records, constants |
| `top.vhdl.j2` | `rtl/{name}.vhd` — top-level entity |
| `core.vhdl.j2` | `rtl/{name}_core.vhd` — bus-agnostic core (unmanaged) |
| `bus_axil.vhdl.j2` | `rtl/{name}_axil.vhd` — AXI4-Lite bus wrapper |
| `bus_avmm.vhdl.j2` | `rtl/{name}_avmm.vhd` — Avalon-MM bus wrapper |
| `register_file.vhdl.j2` | `rtl/{name}_regs.vhd` — standalone register bank |
| `cocotb_test.py.j2` | `tb/{name}_test.py` — cocotb testbench |
| `cocotb_makefile.j2` | `tb/Makefile` — simulation Makefile |
| `regmap_docs.md.j2` | `docs/{name}_regmap.md` — register map documentation |
| `intel_hw_tcl.j2` | `intel/{name}_hw.tcl` — Platform Designer component |
| `xilinx_component_xml.j2` | `xilinx/component.xml` — IP-XACT descriptor |
| `xilinx_package_ip_tcl.j2` | `xilinx/package_ip.tcl` — Vivado IP packaging script |
| `xilinx_xgui.j2` | `xilinx/xgui/{name}_v*.tcl` — Vivado GUI definition |

## Using a Custom Template Directory

Pass `--template-dir` (or `--methodology`) to override selected templates:

```bash
ipcraft generate my_core.ip.yml --template-dir ./my-methodology
```

The generator uses **layered template loading**: it searches your custom
directory first, then falls back to the built-in templates for any file not
present. This means you only need to supply the specific templates you want to
change.

You can specify `--template-dir` multiple times to stack directories:

```bash
ipcraft generate my_core.ip.yml \
  --template-dir ./company-style \
  --template-dir ./project-overrides
```

Directories are searched left to right; the first match wins.

### Python API

```python
from ipcraft.generator.hdl import IpCoreProjectGenerator

# Single custom directory
gen = IpCoreProjectGenerator(template_dir="./my-methodology")

# Multiple directories (searched in order)
gen = IpCoreProjectGenerator(template_dir=["./company-style", "./project-overrides"])
```

## Writing a Custom Template

### Directory Layout

Create a directory with the same filenames as the built-in templates you want
to override. You only need to include files you are changing:

```
my-methodology/
  package.vhdl.j2       # Override the VHDL package template
  cocotb_test.py.j2     # Override the testbench template
  # All other templates fall back to built-in defaults
```

### Discovering the Template Context

Every template receives an `ip_core`-derived context dictionary. Use
`--dump-context` to export the full context to a JSON file so you can see
exactly what variables are available:

```bash
ipcraft generate my_core.ip.yml --dump-context
# Writes template_context.json in the current directory
```

Or programmatically:

```python
from ipcraft.generator.hdl import IpCoreProjectGenerator
from ipcraft.parser import YamlIpCoreParser

ip_core = YamlIpCoreParser().parse_file("my_core.ip.yml")
gen = IpCoreProjectGenerator()
json_str = gen.dump_context(ip_core, bus_type="axil")
print(json_str)
```

### Key Context Variables

| Variable | Type | Description |
|----------|------|-------------|
| `ip_core` | `IpCore` | Full IP core model |
| `name` | `str` | IP core name (lowercase) |
| `vlnv` | `VLNV` | Version/vendor metadata |
| `clocks` | `list` | Clock definitions |
| `resets` | `list` | Reset definitions |
| `ports` | `list` | User-defined ports |
| `bus_interfaces` | `list` | Bus interface definitions |
| `registers` | `list` | Expanded register definitions with computed offsets |
| `has_wstrb` | `bool` | Whether the AXI bus exposes a WSTRB port |
| `bus_type` | `str` | `"axil"` or `"avmm"` |
| `secondary_bus_ports` | `list` | Non-primary bus ports (e.g. AXI-Stream alongside AXI-Lite control) |
| `C_ADDR_WIDTH` | `int` | Address bus width, auto-calculated from register map |

Use `--dump-context` to see the full list for your specific IP core.

### Template Example

```jinja
{# my-methodology/package.vhdl.j2 #}
-- =============================================================================
-- {{ name | upper }}_PKG — auto-generated by IPCraft (custom methodology)
-- =============================================================================
library ieee;
use ieee.std_logic_1164.all;

package {{ name }}_pkg is
{% for reg in registers %}
  constant C_{{ reg.name | upper }}_OFFSET : integer := {{ reg.offset }};
{% endfor %}
end package {{ name }}_pkg;
```

## Registering a Methodology

For large projects, you can install a custom methodology as a Python package
and reference it by importable path instead of a filesystem directory:

```python
import my_company_ipcraft_methodology

gen = IpCoreProjectGenerator(
    template_dir=my_company_ipcraft_methodology.template_dir()
)
```

This allows the methodology to be versioned and distributed independently of
both IPCraft and individual IP core repositories.
